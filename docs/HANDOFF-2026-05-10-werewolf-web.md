## Current Goal

This project is a standalone Werewolf web game room. It no longer depends on Matrix rooms as the actual game room. Matrix is only used for bearer-token authentication and as the source room context when the game is created from a Matrix card.

The intended architecture is:

- Web client renders one standalone game room UI.
- API exposes game creation, joining, leaving, start, room projection, SSE, and runtime tick endpoints.
- Game runtime owns phase progression and game legality.
- Agent runtime is called only when an AI/agent player needs to act.
- The frontend should render server projection and legal targets; it should not decide game legality.

## Important Product Decisions

- There are two room ids:
  - `sourceMatrixRoomId`: the Matrix room that launched or references the game.
  - `gameRoomId`: the actual Werewolf game room id.
- Different Matrix rooms may link to the same `gameRoomId`.
- Creator is also a player. There is no human GM. GM behavior belongs to game runtime.
- Permissions are intentionally simple:
  - creator can start, end, pause, resume, and run runtime progression.
  - other users are participants.
- Users join with a Matrix bearer token. There is no separate `auth/matrix/session`.
- All players can be real users. Creator can fill seats with AI players from a source Matrix room.
- Player join/leave is room state, not game content event.
- Game content is event-based. After game end, GM/agent summary should be written to timeline by runtime, not manually generated by UI.

## Rule Flow Expected

Supported player count: 6 to 12.

Night order:

1. Guard protects.
2. Wolves discuss privately.
3. Wolves vote for kill target.
4. Witch heal.
5. Witch poison.
6. Seer inspects.
7. Night resolution.

Day order:

1. Day speech by seat order.
2. Vote exile.
3. If tie, tied players speak.
4. Revote only among tied candidates.
5. If still tied or timeout rules say no result, no exile.

Important anti-leak rule:

Dead role phases should still appear at night. If the role is absent/dead, runtime waits for the configured short deadline and then skips. The phase itself must not disappear, otherwise players can infer who is alive.

## What Works

The runtime tick path has been verified with real Matrix bot tokens and the Unseal agent LLM endpoint.

Observed verified flow:

- create game
- join 6 kimigame bot users
- start game
- run runtime ticks through night
- guard/wolf/witch/seer agent tool calls are accepted by game runtime
- wolf discussion appears as speech events
- wolf vote resolves target or no-kill by tie rule
- seer result is produced as private event
- night resolves
- day speeches run in order
- vote events are generated
- exile/game-end events are produced

The script expects 6 kimigame tokens by default:

```bash
pnpm integration:real-runtime
```

It prints a proof chain of agent LLM calls, tool calls, accepted runtime events, vote/speech events, and final winner.

## Known Gaps And Next Work

### 1. Manual Player Actions Are Not Fully Wired

The current web UI can select a target, but `onConfirmTarget` in `apps/web/src/routes/game.$gameRoomId.tsx` still only clears selection and shows a demo message:

`当前版本为自动推进流程，确认目标仅用于演示。`

There are no API endpoints yet for a human player to submit:

- guard action
- wolf kill vote
- witch heal/poison
- seer inspect
- day speech
- exile vote

Runtime-driven agent actions work through `POST /games/:gameRoomId/runtime/tick`.

Recommended next backend task:

- Add participant action endpoints, probably:
  - `POST /games/:gameRoomId/actions/night`
  - `POST /games/:gameRoomId/actions/speech`
  - `POST /games/:gameRoomId/actions/vote`
- Validate against runtime projection and legal target set.
- Record the same event types runtime currently records.
- Let runtime tick advance once all required human/agent actions are present or deadline expires.

Recommended next frontend task:

- Replace `onConfirmTarget` demo behavior with the new action endpoints.
- Only show action controls for current user's legal action.
- Prefer server-provided legal target ids over frontend-derived rules.

Important distinction:

- `@kimigame1:keepsecret.io` is a Matrix user id.
- It is not a bearer token.
- The bearer token is the `access_token` column in Synapse DB table `olm_accounts`.

Manual token lookup pattern:

```bash
ssh topsecret-test
psql -h 127.0.0.1 -p 6432 -U keepsecret -d synapse
select bot_user_id, access_token
from olm_accounts
where bot_user_id like '@kimigame%:keepsecret.io'
order by bot_user_id;
```

Verify one token:

```bash
curl -H "authorization: Bearer <ACCESS_TOKEN>" \
  https://keepsecret.io/_matrix/client/v3/account/whoami
```

Expected result includes:

```json
{"user_id":"@kimigame1:keepsecret.io"}
```

## Create A Game From Browser

Open create page with a real Matrix access token:

```text
http://localhost:5173/?token=<URL_ENCODED_ACCESS_TOKEN>
```

Do not pass `@kimigame1:keepsecret.io` as the token. That is the user id and will fail with:

```text
Matrix whoami failed: HTTP 401
```

The create form currently supports:

- Matrix token
- title
- source Matrix room id
- player count
- language

After creating a room, the generated link includes:

```text
?gameRoomId=<GAME_ROOM_ID>&token=<URL_ENCODED_ACCESS_TOKEN>
```

## API Quick Reference

All API requests use:

```http
authorization: Bearer <MATRIX_ACCESS_TOKEN>
content-type: application/json
```

Create game:

```bash
curl -X POST http://localhost:3000/games \
  -H "authorization: Bearer <ACCESS_TOKEN>" \
  -H "content-type: application/json" \
  --data '{
    "sourceMatrixRoomId":"!FWTlpFYoOXfndnfReT:keepsecret.io",
    "title":"demo-room",
    "targetPlayerCount":12,
    "language":"zh-CN",
    "timing":{"nightActionSeconds":45,"speechSeconds":60,"voteSeconds":30},
    "allowedSourceMatrixRoomIds":[],
    "agentSourceMatrixRoomId":"!FWTlpFYoOXfndnfReT:keepsecret.io"
  }'
```

Join game:

```bash
curl -X POST http://localhost:3000/games/<GAME_ROOM_ID>/join \
  -H "authorization: Bearer <ACCESS_TOKEN>"
```

Start game:

```bash
curl -X POST http://localhost:3000/games/<GAME_ROOM_ID>/start \
  -H "authorization: Bearer <CREATOR_ACCESS_TOKEN>"
```

Runtime tick:

```bash
curl -X POST http://localhost:3000/games/<GAME_ROOM_ID>/runtime/tick \
  -H "authorization: Bearer <CREATOR_ACCESS_TOKEN>" \
  -H "content-type: application/json" \
  --data '{
    "agentApiBaseUrl":"https://un-server.dev-excel-alt.pagepeek.org/api",
    "agentApiKey":"<UNSEAL_AGENT_ADMIN_TOKEN>",
    "agentTimeoutMs":15000
  }'
```

Room projection:

```bash
curl http://localhost:3000/games/<GAME_ROOM_ID> \
  -H "authorization: Bearer <ACCESS_TOKEN>"
```

## Handoff Advice

The fastest path to a convincing 12-player browser demo is:

1. Provision or locate tokens for 12 real/agent Matrix users.
2. Add human action submission endpoints.
3. Wire `kimigame1` UI confirmation to real action submission.
4. Add an autoplay/dev-only stage driver only if needed for presentation, but keep it clearly separate from production runtime.
5. Add cinematic phase animations after the action loop is real.

Do not judge the runtime by the current visual UI alone. The backend agent-driven runtime is further along than the browser interaction layer. The main product risk is now connecting the elegant UI to real per-player action endpoints without moving legality back into the client.
